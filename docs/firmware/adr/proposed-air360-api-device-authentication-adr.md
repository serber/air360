# Air360 API Device Authentication ADR

## Status

Proposed.

## Decision Summary

Protect Air360 API measurement uploads with a device-specific shared secret and HMAC-SHA256 request signatures. The firmware stores an `upload_secret` received during provisioning, signs every Air360 API upload request, and includes timestamp/nonce headers to prevent replay. The backend accepts unsigned upload requests only during a migration window.

The design does not require user accounts or a customer portal. First registration can use trust-on-first-use (TOFU). Recovery after a full device erase uses a short-lived local pairing code shown by the device web UI, so knowing a `device_id` alone is not enough to rotate the upload secret.

## Context

The current Air360 API integration identifies the device by `device_id` in the URL:

```http
PUT /v1/devices/{device_id}/batches/{batch_id}
```

The adapter also registers the device once per boot:

```http
PUT /v1/devices/{device_id}/register
```

This is enough for routing data to a device record, but it is not authentication. Anyone who learns a device ID can construct upload requests for that device unless the backend requires proof that the sender knows a device-specific secret.

The project does not currently plan a user account area or an operator admin panel for device claim/reset workflows. The recovery path therefore must work from the device's local web UI plus backend public endpoints.

## Goals

- Prevent measurement spoofing by clients that only know `device_id`.
- Keep the firmware/backend protocol simple enough for ESP-IDF C++ firmware.
- Avoid requiring user accounts, a customer portal, or a manual backend admin flow.
- Allow a device to recover after `erase-flash` or NVS loss.
- Preserve at-least-once upload semantics and existing upload queue behavior.
- Make replayed upload requests rejectable by the backend.

## Non-Goals

- Protect against an attacker with physical possession of the device and flash extraction capability.
- Implement secure boot, flash encryption, or eFuse key storage as part of this ADR.
- Add public/private key provisioning in the first version.
- Authenticate Sensor.Community, InfluxDB, or Custom Upload backends.
- Encrypt payload contents beyond HTTPS transport encryption.

## Threat Model

### Protected Against

- A remote client that knows `device_id` and fabricates sensor batches.
- A remote client that replays an old valid upload request.
- A remote client that tries to rotate an existing device's upload secret using only `device_id`.

### Not Protected Against

- Physical extraction of an unencrypted NVS `upload_secret`.
- Malware or compromised firmware running on the device.
- First-registration hijack during TOFU if an attacker registers a device ID before the real device.
- TLS compromise or backend private key compromise.

## Architectural Decision

### 1. Device Secret

The backend stores one active `upload_secret` per Air360 device. The firmware stores the same secret in NVS.

Properties:

- 32 random bytes generated by the backend.
- Encoded as base64url without padding in JSON responses.
- Never returned by the backend after initial provisioning or explicit pairing recovery.
- Rotating the secret invalidates the previous secret immediately unless the backend enables a short migration grace window.

Firmware storage:

- Add an Air360 API credential record in NVS, separate from the editable backend config.
- Store the secret bytes, key version, and provisioning state.
- Do not expose the secret in status JSON, logs, web UI, BLE, or diagnostics dumps.
- Status may expose only booleans such as `air360_api.provisioned` and `air360_api.secret_version`.

### 2. Registration Modes

The backend supports three registration outcomes:

1. New device: create device record and return `upload_secret`.
2. Existing provisioned device: do not return `upload_secret`; report that the device is already registered.
3. Existing device in pairing recovery: rotate secret and return the new `upload_secret`.

This allows simple first boot while preventing a client that only knows `device_id` from fetching the active secret.

### 3. First Registration: TOFU

For a new device ID that does not exist on the backend, the firmware calls registration and receives the first secret.

Request:

```http
PUT /v1/devices/{device_id}/register
Content-Type: application/json
User-Agent: air360/{firmware_version}
```

```json
{
  "schema_version": 1,
  "device": {
    "device_id": "123456789012",
    "short_device_id": "789012",
    "esp_mac_id": "aabbccddeeff",
    "name": "air360",
    "board_name": "esp32-s3-devkitc-1",
    "firmware_version": "0.1.0"
  },
  "location": {
    "latitude": 55.751244,
    "longitude": 37.618423
  }
}
```

Successful response for a new device:

```http
201 Created
Content-Type: application/json
Cache-Control: no-store
```

```json
{
  "schema_version": 1,
  "status": "registered",
  "device_id": "123456789012",
  "secret_version": 1,
  "upload_secret": "base64url-encoded-32-byte-secret",
  "signature": {
    "algorithm": "HMAC-SHA256",
    "timestamp_tolerance_seconds": 300,
    "nonce_ttl_seconds": 600
  }
}
```

Response for an already provisioned device:

```http
409 Conflict
Content-Type: application/json
Cache-Control: no-store
```

```json
{
  "schema_version": 1,
  "status": "already_registered",
  "device_id": "123456789012",
  "secret_required": true,
  "recovery": {
    "method": "local_pairing_code",
    "start_endpoint": "/v1/devices/123456789012/pairing/start"
  }
}
```

Firmware behavior:

- If `registered`: store `upload_secret` and `secret_version`, then upload normally.
- If `already_registered` and local secret exists: keep using the local secret.
- If `already_registered` and local secret is missing: enter `secret_missing` state and expose a local web UI action to start pairing recovery.

### 4. Recovery Without User Accounts: Local Pairing Code

Recovery is needed after full flash erase, NVS corruption, or device replacement. The backend has no user account flow, so recovery relies on a pairing code shown by the local device web UI.

The pairing code proves local access to the device UI. It does not prove legal ownership, but it prevents remote secret rotation using only `device_id`.

Firmware pairing flow:

1. Device detects that Air360 API is configured but `upload_secret` is missing.
2. User opens the device local web UI.
3. User clicks "Pair Air360 API" on the Backends page.
4. Firmware generates:
   - `pairing_code`: 8-10 decimal or Crockford Base32 characters.
   - `pairing_nonce`: 16 random bytes, base64url encoded.
   - `expires_at_uptime_ms`: now + 10 minutes.
5. Firmware displays the pairing code locally.
6. Firmware calls backend `pairing/start`.
7. User opens the backend pairing URL and enters the displayed code.
8. Firmware polls `pairing/status`.
9. Backend returns a rotated `upload_secret` to the device after successful confirmation.

#### 4a. Start Pairing

Request from firmware:

```http
POST /v1/devices/{device_id}/pairing/start
Content-Type: application/json
User-Agent: air360/{firmware_version}
```

```json
{
  "schema_version": 1,
  "device_id": "123456789012",
  "pairing_nonce": "base64url-random-16-bytes",
  "pairing_code_hash": "base64url(sha256(pairing_code + \":\" + pairing_nonce))",
  "expires_in_seconds": 600,
  "firmware_version": "0.1.0"
}
```

Successful response:

```http
202 Accepted
Content-Type: application/json
Cache-Control: no-store
```

```json
{
  "schema_version": 1,
  "status": "pending",
  "pairing_session_id": "base64url-random-session-id",
  "pairing_url": "https://api.air360.ru/v1/pair",
  "poll_after_seconds": 2,
  "expires_in_seconds": 600
}
```

Backend requirements:

- Store only `pairing_code_hash`, never the raw code.
- Expire the session after `expires_in_seconds`.
- Allow only one active pairing session per device; starting a new one invalidates the previous session.
- Rate-limit by device ID and source IP.

#### 4b. Human Confirmation Page/API

The backend exposes a minimal public pairing page or API. No user account is required.

Browser page:

```http
GET /v1/pair
```

The page asks for:

- `device_id`
- pairing code shown in the local device UI

Confirmation API used by that page:

```http
POST /v1/devices/{device_id}/pairing/confirm
Content-Type: application/json
```

```json
{
  "schema_version": 1,
  "pairing_session_id": "base64url-random-session-id",
  "pairing_code": "A1B2C3D4"
}
```

Backend behavior:

- Look up the active pending session for `device_id`.
- Recompute `sha256(pairing_code + \":\" + pairing_nonce)`.
- Compare with stored `pairing_code_hash` using constant-time comparison.
- On success, mark the session confirmed and create a new 32-byte `upload_secret`.
- Do not return the secret to the browser.

Successful confirmation response:

```http
200 OK
Content-Type: application/json
Cache-Control: no-store
```

```json
{
  "schema_version": 1,
  "status": "confirmed",
  "device_id": "123456789012"
}
```


Pending response:

```http

```

```json

```

Confirmed response:

```http
200 OK
Content-Type: application/json
Cache-Control: no-store
```

```json
{
  "schema_version": 1,
  "status": "confirmed",
  "device_id": "123456789012",

    "algorithm": "HMAC-SHA256",

  }
}
```

Firmware behavior:

- Store the new secret and version atomically in NVS.
- Clear the local pairing code and nonce.
- Resume signed uploads.

### 5. Signed Upload Requests

Every Air360 API data upload includes HMAC headers.

Request:

```http
PUT /v1/devices/{device_id}/batches/{batch_id}
Content-Type: application/json
User-Agent: air360/{firmware_version}
X-Air360-Device: {device_id}
X-Air360-Key-Version: {secret_version}
X-Air360-Timestamp: {unix_ms}
X-Air360-Nonce: {base64url-random-16-bytes}
X-Air360-Body-SHA256: {base64url(sha256(body))}
X-Air360-Signature: {base64url(hmac_sha256(upload_secret, canonical_request))}
```

Canonical request string:

```text
AIR360-HMAC-SHA256
{METHOD}
{PATH_WITH_QUERY}
{X-Air360-Device}
{X-Air360-Key-Version}
{X-Air360-Timestamp}
{X-Air360-Nonce}
{X-Air360-Body-SHA256}
```

Rules:

- `METHOD` is uppercase, e.g. `PUT`.
- `PATH_WITH_QUERY` starts with `/` and excludes scheme, host, and fragment.
- Query parameters, if any, must be in the exact transmitted order. The current batch upload URL has no query string.
- Header names are case-insensitive on the wire, but canonical fields are exactly the values above.
- Body hash is computed over the exact UTF-8 bytes transmitted.
- Signature is HMAC-SHA256 over the canonical request bytes using the raw 32-byte secret.
- Encodings use base64url without padding.

Example canonical request:

```text
AIR360-HMAC-SHA256
PUT
/v1/devices/123456789012/batches/1744400000000
123456789012
1
1744400000123
7lT7WqRUcucbLioY6uUuwA
QZzQ1H8zM0aXQ2RZxEXAMPLEBODYHASH
```

Backend verification:

1. Find device by `X-Air360-Device` and path `{device_id}`; both must match.
2. Find active secret by `X-Air360-Key-Version`.
3. Reject if timestamp is outside tolerance, default 300 seconds.
4. Reject if nonce was already used for this device/key version inside `nonce_ttl_seconds`.
5. Recompute body SHA-256 and compare to `X-Air360-Body-SHA256`.
6. Recompute HMAC and compare using constant-time comparison.
7. Process batch only after signature verification succeeds.
8. Store nonce until TTL expires.

Replay behavior:

- Reusing the same nonce with the same key version returns `409 Conflict` or `401 Unauthorized`.
- Re-sending the same `batch_id` with a new nonce and valid signature may return existing idempotency behavior, currently success for duplicate `409` in firmware classification.

### 6. Backend Error Contract

Registration and pairing errors:

| HTTP | `code` | Meaning | Firmware action |
|------|--------|---------|-----------------|
| 400 | `bad_request` | Invalid JSON, missing field, malformed base64url | Show config/protocol error |
| 401 | `invalid_signature` | Signed endpoint failed HMAC verification | Keep queue window; report auth error |
| 403 | `pairing_required` | Existing device requires pairing before secret can be issued | Enter secret-missing/pairing state |
| 404 | `device_not_found` | Unknown device where TOFU is disabled | Show registration error |
| 409 | `already_registered` | Device exists and backend will not return secret | Use local secret or enter recovery |
| 409 | `nonce_replay` | Upload nonce already used | Generate a new nonce and retry later |
| 410 | `pairing_expired` | Pairing session expired | Clear session; user can start again |
| 429 | `rate_limited` | Too many attempts | Respect `Retry-After` |
| 500 | `server_error` | Backend failure | Retry using normal upload policy |

Error body:

```json
{
  "schema_version": 1,
  "status": "error",
  "code": "already_registered",
  "message": "Device is already registered; local pairing is required to rotate the upload secret.",
  "retry_after_seconds": 60
}
```

### 7. Firmware State Machine

Air360 API adapter provisioning states:

| State | Meaning | Upload behavior |
|-------|---------|-----------------|
| `unprovisioned` | No local secret; backend registration not attempted or failed | Try registration before upload |
| `provisioned` | Local secret exists | Sign uploads |
| `secret_missing` | Backend says already registered but local secret is absent | Do not upload; expose pairing action |
| `pairing_pending` | Local pairing code active; waiting for backend confirmation | Poll status; do not upload |
| `pairing_failed` | Pairing expired or failed | User may restart pairing |

`UploadManager` queue semantics remain unchanged:

- Failed auth keeps the current backend window for retry.
- `already_registered` with missing secret is a backend-specific configuration/auth failure.
- Other backends continue uploading independently.
- Best-effort demotion still applies after repeated backend-specific failures.

### 8. Migration Plan

Phase 1: Backend accepts both unsigned and signed Air360 API uploads.

- Registration returns `upload_secret` for new devices.
- Firmware stores secret and starts signing.
- Backend records whether a device has ever uploaded with a valid signature.

Phase 2: Backend requires signatures for devices with an active secret.

- Unsigned upload for provisioned devices returns `401 invalid_signature`.
- Legacy devices without a secret can still register.

Phase 3: Backend disables unsigned uploads globally.

- All Air360 API uploads require HMAC headers.
- Recovery uses local pairing.

## API Summary

| Endpoint | Method | Auth | Purpose |
|----------|--------|------|---------|
| `/v1/devices/{device_id}/register` | `PUT` | none for TOFU | Create device and issue first secret, or report already registered |
| `/v1/devices/{device_id}/pairing/start` | `POST` | local pairing hash | Start secret recovery session |
| `/v1/pair` | `GET` | none | Minimal human confirmation page |
| `/v1/devices/{device_id}/pairing/confirm` | `POST` | pairing code | Confirm local access and rotate secret |
| `/v1/devices/{device_id}/pairing/status` | `POST` | session id + nonce | Device polls for new secret |
| `/v1/devices/{device_id}/batches/{batch_id}` | `PUT` | HMAC headers | Upload signed measurement batch |

## Open Questions

- Should the firmware generate the initial secret and send it to the backend, or should the backend generate it? This ADR chooses backend-generated secrets for simpler backend rotation.
- Should signed requests use `unix_ms` or seconds? This ADR uses milliseconds because firmware already tracks upload timestamps in milliseconds.
- Should pairing confirmation be a backend web page or JSON-only API? The contract supports both; a minimal backend page is friendlier without requiring accounts.
- Should NVS encryption or flash encryption become mandatory before production? This ADR does not require it, but production devices should enable stronger local secret protection.

## Practical Conclusion

Implement HMAC-SHA256 signed Air360 API uploads with a backend-issued per-device secret. Use TOFU for first registration and local pairing code recovery for erased devices. This blocks data spoofing by clients that only know `device_id` without introducing user accounts or an operator portal.
